how2heap house_of_orange

利用unsorted bin attack覆盖_io_list_all指向可控内存，并在其中构造_IO_FILE_plus结构体来控制流程

源码(省去注释):

流程 part1：unsoretd bin attack

  • p1 = malloc(0x400-16)，假设p1存在溢出，且可以覆盖top chunk的size
  • top[1] = 0xc01，覆盖size，使得top_head + size在内存页对齐(一个内存页大小为4K，即0x1000，所以top_head+size要为0x1000的倍数)
  • p2 = malloc(0x1000)，malloc的大小需大于top chunk的size，使得能够触发sbrk扩展出new_top，p2则位于new top_chunk中
  • old top_chunk剩余的部分则被free添加到unsorted bin中，同时其fd，bk也更新为指向

流程 part2:_IO_FILE attack(FSOP)

  • io_list_all = bk(main_arena+88)+0x9a8，io_list_all位于main_arena+88+0x9a8
  • top[3] = io_list_all - 0x10，利用溢出覆盖bk为_IO_list_all-0x10
  • memcpy( ( char *) top, “/bin/sh\x00”, 8)，在old top_chunk头部填充’/bin/sh’作为system的参数
  • top[1] = 0x61，利用溢出覆盖top的size为0x61，此时fake _IO_FILE struct已经准备好了，开始向其中某些成员变量填充值以完成最后的attack

分析&理解：

part1

part1主要进行内存泄露，为part2的FSOP做准备。

一般进行内存泄露会想到用free一个非fastbin来泄露fd，bk上的main_arena从而得到libc base。但当程序不提供free时，part1提供了另外一种思路，在没有free的情况下也能调用free。

我们知道，用户申请的chunk来自于top chunk的分裂，而top chunk的初始大小是一定的(0x21000)，所以随着申请的chunk越来越多，top chunk剩下的部分会越来越少，当剩余部分小于用户请求的大小时会调用sysmalloc()。

并分为两种情况：1. 申请大小小于mp_.mmap_threshold[1][2][3]，则调用brk从old top_chunk尾部向后扩展出new top_chunk。2. 申请大小大于mp_.mmap_threshold，则调用mmap()。

为了调用_int_free，则需要进入第一种情况：new top_chunk生成后，对old top_chunk调用_int_free，然后被添加到unsorted bin，fd和bk被设置为main_arena+88，从而泄露libc base。

在这之前还有一些检查：

所以，构造的top_chunk size还要满足一下条件：

  • size大于MINSIZE且小于需求的大小
  • prev_inuse = 1
  • top_chunk + size满足页对齐

patr2

IO_FILE相关知识：

https://ctf-wiki.github.io/ctf-wiki/pwn/linux/io_file/introduction-zh/

part2进行FSOP，布置攻击链，最后用malloc(10)触发。

FSOP的思想是伪造_IO_FILE结构体和vtable，利用方式是使malloc报错，从而触发函数调用链：malloc_printerr -> _libc_message -> abort -> _IO_flush_all_lockp，_IO_flush_all_lockp会从_IO_list_all指向的第一个IO_FILE_plus结构体开始遍历，为每一个结构体调用其vtable中的_IO_overflow。因此，只要劫持IO_list_all，使得IO_FILE_ chain中有我们伪造的结构体，接着将fake vtable中的_IO_overflow替换为system，再在合适的位置传递参数’/bin/sh’，并绕过一些检查，在触发报错的同时就可以拿到shell。

所以现在需要找到一块可控的内存以伪造_IO_FILE_plus：

利用part1泄露的main_arena，计算出_io_file_list的位置，然后就能将old top_chunk的bk覆盖为_io_file_list-0x10。进行malloc时，由于old top_chunk不属于last reminder，所以malloc(10)不会让old top_chunk分裂返回给用户，而是将其添加到对应大小的bin中。因为要将unsorted bin中的chunk放入small/large bin，所以先要将其从unsorted bin中取出来，至此触发攻击链的第一步：利用取出操作将io_list_all指向main_arena+88。

现在已经控制_IO_list_all指向了main_arena+88，如果我们能在main_arena中布置fake _IO_FILE_plus就可以控制程序流，但事实上很难控制main_arena中的值。

由于main_arena+88被_IO_list_all视为链表中的(首个)_IO_FILE_plus结构体，_IO_FILE中*_chain所处的偏移为0x68，而恰好在以main_arena+88为起始的偏移为0x68处对应的是大小为0x60的smallbin(即smallbin[4])的bk，也就是说如果我们将old top_chunk size覆盖为0x61，在malloc时，他就会被归类到smallbin[4]，正好是_chain所指向的下一个_IO_FILE结构体。所以只要在old top_chunk中布置好各种参数，在_IO_flush_all_lockp遍历链表时自然会执行到我们的攻击链上。

因为old top_chunk的大小被修改后属于small bin，所以计算该chunk的大小属于small bin的哪一个index，然后将bck和fwd设置为指向对应的small bin index

紧接着就是将其放入small bin中，所以small bin[5] (index=1的small bin为0x30字节，所以index=4为0x60字节)指向了old top_chunk。

分析函数

题目提供了三个功能：申请chunk、修改chunk和查看chunk。没有释放chunk

build：

若全局变量 build_count > 3则提示已满并exit

新建一个house，一共会创建三个chunk，第一个和第三个chunk的大小固定为0x20，第二个大小为用户自定义且最大为0x1000。

第一个chunk包含的信息：两个指针，第一个指向第三个chunk，第二个指向第二个chunk
第二个chunk包含的信息：用户自定内容，不可溢出
第三个chunk包含的信息：前8字节分为两个dword，低位存price，高位存color+30

同时更新全局变量：

解题过程

build第一个house并upgrade，覆盖top chunk的size，使其满足：size小于0x1000，&top chunk+size页对齐，previnuse位为1

bulid第二个house，大小为0x1000，触发sysmalloc和_int_free，使得old top_chunk被添加到unsorted bin，同时fd，bk被设置

bulid第三个house，大小为large bin，此时填写的name位于之前old top_chunk的fd处，所以填充7个字节(加上换行共8字节)，调用see，可以泄露出main_arena，根据偏移计算出libc base。再次upgrade填充0xf字节，泄露fd_nextsize处的值，该值指向malloc该chunk前，old top_chunk head处的地址，根据偏移计算出heap base。

申请第三个house时，会在large bin中搜索大小满足的chunk，由于large bin中没有chunk，所以把unsorted bin中的old top_chunk放入large bin中对应大小的index下(这里的index为98)，并设置好fd，bk，fd_nextsize和bk_nextsize；fd和bk分别指向main_arena中对应index的bin head；
